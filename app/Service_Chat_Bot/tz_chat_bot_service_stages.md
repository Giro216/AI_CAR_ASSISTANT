# Техническое задание: `chat-bot-service`

## 1. Назначение сервиса

`chat-bot-service` — это orchestration-сервис для AI-подбора автомобилей. Сервис принимает пользовательские сообщения, хранит диалог, формирует контекст для LLM, вызывает OpenRouter / GLM 4.5 Air, управляет tool calling, обращается к смежным микросервисам и возвращает пользователю структурированный ответ.

### Что сервис делает
- принимает сообщения от Gateway;
- загружает историю диалога;
- формирует prompt и контекст для LLM;
- вызывает OpenRouter;
- обрабатывает tool calls;
- при необходимости обращается к другим сервисам;
- хранит и обновляет историю диалога;
- постепенно формирует профиль пользователя;
- возвращает ответ в frontend через Gateway.

### Что сервис не делает
- не хранит каталог автомобилей;
- не занимается аутентификацией;
- не реализует бизнес-логику каталога;
- не считает сложный ranking в первой версии;
- не является frontend-сервисом.

---

## 2. Текущие ограничения проекта

На текущем этапе:
- backend уже есть на FastAPI + Pydantic;
- frontend уже есть на React;
- связь идёт через Gateway;
- отдельный `profile-service` пока **не реализован**;
- отдельный `ranking-service` пока **не нужен**;
- основная LLM — **GLM 4.5 Air** через OpenRouter.

### Решение для MVP
Профиль пользователя хранить **внутри `chat-bot-service`**, а не выносить в отдельный микросервис.

---

## 3. Цели по этапам

### Этап 1. Базовый чат
Цель: получить рабочий чат без памяти, профиля и внешних инструментов.

### Этап 2. Память диалога
Цель: сохранять историю сообщений и подмешивать её в контекст.

### Этап 3. Профиль пользователя
Цель: начинать хранить предпочтения клиента внутри `chat-bot-service`.

### Этап 4. Tool calling и каталог
Цель: подключить поиск машин через `catalog-service`.

### Этап 5. Улучшение качества
Цель: добавить streaming, summary, retry, validation, monitoring.

### Этап 6. Вынос в отдельные сервисы
Цель: при необходимости отделить профиль и ranking в отдельные микросервисы.

---

# 4. Этап 1 — Базовый рабочий чат

## 4.1 Цель
Сделать минимальную работающую цепочку:

`Frontend -> Gateway -> chat-bot-service -> OpenRouter -> ответ`

## 4.2 Задачи

### 4.2.1 Создать FastAPI-приложение
**Задача:** поднять отдельный сервис `chat-bot-service`.

**Нужно:**
- FastAPI app;
- `/health` endpoint;
- базовая структура папок;
- конфигурация через env;
- логирование.

**Результат:** сервис запускается в Docker и отвечает на healthcheck.

---

### 4.2.2 Подключить OpenRouter
**Задача:** научить сервис отправлять запрос в GLM 4.5 Air.

**Нужно:**
- клиент OpenRouter;
- API key через env;
- базовый вызов chat completions;
- обработка ответа.

**Результат:** сервис получает ответ от модели и возвращает его наружу.

---

### 4.2.3 Создать endpoint чата
**Задача:** добавить endpoint для одного сообщения пользователя.

**Нужно:**
- `POST /chat/message`;
- входная схема `ChatMessageIn`;
- выходная схема `ChatMessageOut`.

**Результат:** фронтенд может отправить сообщение и получить ответ.

---

### 4.2.4 Добавить system prompt
**Задача:** задать модели роль AI-консультанта по подбору авто.

**Нужно:**
- prompt с правилами ответа;
- prompt с инструкцией задавать уточняющие вопросы;
- prompt с запретом на выдумывание данных.

**Результат:** модель отвечает в нужной роли.

## 4.3 Что не делать на этапе 1
- не хранить историю сообщений;
- не делать profile extraction;
- не вызывать catalog-service;
- не использовать Redis;
- не использовать PostgreSQL;
- не использовать ranking.

## 4.4 Критерий готовности этапа 1
- сервис поднят;
- OpenRouter отвечает;
- сообщение из frontend проходит до модели;
- модель возвращает осмысленный ответ.

---

# 5. Этап 2 — Память диалога

## 5.1 Цель
Сделать так, чтобы модель помнила предыдущие сообщения в рамках беседы.

## 5.2 Задачи

### 5.2.1 Создать модель Conversation
**Нужно:**
- `conversation_id`;
- `user_id`;
- `summary`;
- даты создания и обновления.

**Результат:** у каждой беседы есть собственная сущность.

---

### 5.2.2 Создать модель Message
**Нужно:**
- `message_id`;
- `conversation_id`;
- `role` (`user`, `assistant`, `tool`);
- `content`;
- `created_at`.

**Результат:** каждое сообщение сохраняется отдельно.

---

### 5.2.3 Реализовать репозиторий истории
**Нужно:**
- `save_message()`;
- `load_recent_messages()`;
- `save_summary()`.

**Результат:** сервис умеет читать и писать историю чата.

---

### 5.2.4 Подмешивать историю в prompt
**Нужно:**
- при каждом новом запросе загружать последние сообщения;
- ограничивать размер контекста;
- при необходимости использовать summary.

**Результат:** модель видит предыдущий диалог.

## 5.3 Критерий готовности этапа 2
- модель учитывает предыдущее сообщение;
- повторные вопросы уменьшаются;
- история не теряется после одного ответа.

---

# 6. Этап 3 — Профиль клиента внутри `chat-bot-service`

## 6.1 Цель
Начать формировать профиль пользователя без отдельного profile-service.

## 6.2 Решение
Пока профиль хранится в том же сервисе, в отдельной таблице или хранилище внутри `chat-bot-service`.

## 6.3 Задачи

### 6.3.1 Создать модель UserProfile
**Поля для старта:**
- `budget`;
- `country`;
- `family`;
- `children_count`;
- `body_type`;
- `fuel_type`;
- `transmission`;
- `annual_mileage`;
- `city_driving`;
- `sporty`;
- `preferred_brands`;
- `must_have_features`;
- `avoid_features`.

**Результат:** есть структурированная схема профиля.

---

### 6.3.2 Создать хранилище профиля
**Нужно:**
- читать профиль по `user_id`;
- сохранять изменения профиля;
- обновлять отдельные поля, а не всё целиком.

**Результат:** сервис может накапливать предпочтения клиента.

---

### 6.3.3 Добавить extraction-логику
**Нужно:**
- после сообщения пользователя выделять признаки;
- преобразовывать их в JSON-патч;
- сливать патч с текущим профилем.

**Результат:** профиль обновляется автоматически по ходу диалога.

## 6.4 Критерий готовности этапа 3
- сервис умеет извлекать предпочтения;
- профиль обновляется после сообщений;
- ответы становятся более персонализированными.

---

# 7. Этап 4 — Tool calling и `catalog-service`

## 7.1 Цель
Научить чат не только говорить, но и искать конкретные автомобили.

## 7.2 Задачи

### 7.2.1 Создать contract для tool calls
**Инструменты на старте:**
- `search_cars`;
- `compare_cars`;
- `get_user_profile`;
- `update_user_profile`.

**Результат:** LLM может просить сервис выполнить внешнее действие.

---

### 7.2.2 Создать `catalog-service`
**Нужно:**
- отдельный микросервис;
- endpoint поиска автомобилей;
- входная схема поиска;
- выходная схема списка кандидатов.

**Результат:** сервис возвращает список машин по фильтрам.

---

### 7.2.3 Подключить вызовы catalog-service
**Нужно:**
- `chat-bot-service` получает tool call от модели;
- вызывает `catalog-service`;
- получает ответ;
- отправляет результат обратно в LLM.

**Результат:** модель начинает работать с реальными данными каталога.

---

### 7.2.4 Собрать финальный ответ после tool call
**Нужно:**
- передать tool result в модель;
- получить итоговое объяснение;
- вернуть человеку 1–3 рекомендованных варианта.

**Результат:** появляется настоящий AI-подборщик.

## 7.3 Критерий готовности этапа 4
- чат умеет искать машины;
- ответы содержат конкретные модели;
- есть объяснение, почему именно они подходят.

---

# 8. Этап 5 — Улучшенная версия chat-bot-service

## 8.1 Цель
Сделать сервис устойчивым, удобным для пользователей и пригодным для расширения.

## 8.2 Задачи

### 8.2.1 Добавить streaming
**Нужно:**
- SSE или WebSocket;
- частичная отдача текста из модели;
- отображение typing state в React.

**Результат:** пользователь видит ответ по мере генерации.

---

### 8.2.2 Добавить summary старых сообщений
**Нужно:**
- сокращать длинные диалоги;
- хранить summary отдельно;
- не перегружать prompt.

**Результат:** сервис работает с длинными чатами без потери качества.

---

### 8.2.3 Добавить retries и timeout policy
**Нужно:**
- retry при временной ошибке OpenRouter;
- retry при ошибке catalog-service;
- timeout для HTTP-запросов.

**Результат:** сервис не падает от кратких сетевых проблем.

---

### 8.2.4 Добавить Pydantic validation на всех границах
**Нужно:**
- валидация входящих запросов;
- валидация ответов моделей;
- валидация ответов внешних сервисов.

**Результат:** меньше сломанных ответов и меньше скрытых ошибок.

---

### 8.2.5 Добавить trace_id и structured logging
**Нужно:**
- единый `trace_id`;
- логирование запросов и ответов;
- логирование tool calls;
- логирование ошибок.

**Результат:** проще отлаживать сценарии.

## 8.3 Критерий готовности этапа 5
- чат стабильный;
- работает на длинных сессиях;
- ошибки видны и диагностируются;
- UI получает потоковый ответ.

---

# 9. Этап 6 — Вынос сервисов

## 9.1 Цель
Когда MVP уже работает, разделить ответственность на отдельные микросервисы.

## 9.2 Что можно вынести

### 9.2.1 `profile-service`
Когда профиль стабилен и ты понимаешь, какие поля действительно нужны.

### 9.2.2 `ranking-service`
Когда появится достаточно данных, чтобы считать scoring автомобилей отдельно от LLM.

### 9.2.3 `search-service`
Когда нужен внешний поиск по авто, рынку, объявлениям и источникам.

## 9.3 Критерий готовности этапа 6
- логика уже проверена в MVP;
- сервисы выносятся без изменения UX;
- API между сервисами стабилизированы.

---

# 10. Рекомендуемый порядок реализации

## Очерёдность
1. Этап 1 — базовый чат.
2. Этап 2 — память.
3. Этап 3 — профиль внутри chat-bot-service.
4. Этап 4 — catalog-service и tool calling.
5. Этап 5 — streaming, retries, validation, logging.
6. Этап 6 — вынос profile-service и ranking-service.

---

# 11. Definition of Done для MVP

MVP считается готовым, если:
- пользователь может написать сообщение в frontend;
- сообщение проходит через Gateway в `chat-bot-service`;
- сервис вызывает GLM 4.5 Air через OpenRouter;
- ответ возвращается пользователю;
- история диалога сохраняется;
- профиль пользователя начинает наполняться;
- сервис может вызывать `catalog-service`;
- рекомендации содержат конкретные модели машин и пояснение.

---

# 12. Риски и ограничения

- слишком ранний вынос profile-service усложнит MVP;
- без memory модель будет терять контекст;
- без structured validation ответы модели могут быть нестабильны;
- без tool calling чат не станет настоящим автоподборщиком;
- без summary длинные диалоги быстро переполнят context window.

---

# 13. Итог

Для текущего состояния проекта правильный путь такой:
- сначала сделать сильный `chat-bot-service` внутри FastAPI;
- хранить профиль пользователя внутри него;
- подключить OpenRouter и GLM 4.5 Air;
- потом добавить memory;
- потом catalog integration;
- только после этого выносить профиль и ranking в отдельные сервисы.
